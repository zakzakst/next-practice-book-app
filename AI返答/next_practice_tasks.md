# 次回の練習に向けた課題リスト

プロジェクト内の `TODO:` コメントを抽出し、今後の改善点や調査すべき課題としてカテゴリごとに整理しました。次回の練習の参考にしてください。

## 1. テスト関連（単体テスト・E2E）
テスト環境の構築や、UIコンポーネント・フォームのテスト手法に関して多くの課題が残っています。

*   **shadcn UIコンポーネントのテスト:**
    *   `Select` コンポーネントの操作・単体テストの方法にクセがあるため、テスト手法を確立する。（保留中）
*   **React Hook Formのテスト:**
    *   単体テストおよびE2Eテストでのバリデーションテストが上手くいかないため、テストの書き方を調査・改善する。
*   **E2Eテスト (Playwright / Jestのセオリー):**
    *   `afterAll` 的な処理の書き方を調べる。
    *   データベースを更新するテストの後に、データベースを初期化（クリーンアップ）する方法のセオリーを調べる。
    *   未ログイン時に `toHaveURL` が成功せず `page.url()` を利用した件について原因を調べる。
    *   E2Eテストの観点整理（一覧からリンクを探すのではなく、直接詳細ページのURLにアクセスする等、テストの目的に応じた画面遷移を検討）。
*   **Vitestの設定:**
    *   `vitest.config.ts` で拡張機能が上手く動かず追記した記述について、本来の正しい設定方法を調べる。

## 2. API設計・データフェッチ（SWR）
APIのエンドポイント設計や、クライアントでのデータ取得（SWR）の共通化について見直しが必要です。

*   **SWRとデータ取得アーキテクチャ:**
    *   複数APIを走らせる際のエラー/ロードのハンドリングが複雑になる問題の解決。
    *   SWRの共通 `fetcher` （`findOneFetcher` と `findAllFetcher`）の設計が用途と合致しなくなっているため、共通化の方法を見直す。
    *   Server Components (`use server` のファイル) でSWRを使うか（SEO目的等）、またそれが適切かどうかの再考。
    *   データ取得ルールの策定（POSTで取得・条件をすべてリクエストに含める方針の見直し等）。
*   **APIエンドポイント設計:**
    *   書籍詳細とレビューデータのAPIを分けたことによるバケツリレーの解消など、API設計を見直す。
    *   REST APIではないものとの命名規則の区別方法を考える。

## 3. エラーハンドリング・レスポンス設計
APIのレスポンスルールや、クライアント側でのエラー処理に関するTODOが最も多く残っています。

*   **エラーレスポンスのルール化:**
    *   すべてのAPIルート（ログイン、登録、書籍取得、レビュー関連など）で共通のエラーレスポンスのルールを策定する。
    *   `DELETE` メソッドで何も返さない時の正しい書き方を調べる。
*   **クライアント側でのエラー判定:**
    *   現状、APIレスポンスがHTTPエラー（`res.ok`）にならず `res.title` の有無で判定しているような箇所があるため、APIのレスポンス設計を見直し、適切なエラーハンドリングを実装する。
    *   各画面（レビュー作成、レビュー編集、AuthContextなど）で後回しにしているエラー時の処理を実装する。

## 4. UI/UX・フォームの実装
フォームの使い勝手や、UIのブラッシュアップに関する課題です。

*   **フォームの初期値とAPI設計:**

## 5. バックエンド処理・その他
*   **パスワード等の不要なデータ除外:**
    *   Prisma等でユーザーデータを取得する際、毎回 `password` などを手動で消す処理が不便なため、レスポンスから除外する共通処理やセオリー（`roles`を含めるか等）を調べる。
*   **環境変数・命名規則:**
    *   環境変数を用いたURLの出し分け処理の実装。
    *   `AuthContext.tsx` と Hooksの `useAuth.ts` で命名や役割が重複している部分を整理・検討する。
*   **テンプレートへの反映:**
    *   テストや設定ファイル（`eslint.config.mjs` 等）での修正を元のテンプレートリポジトリにも反映させる。
